[ASSISTENTE] - 11/07/2026, 15:19:11
Para listar as notícias (posts) no projeto **notic-ia**, você pode utilizar o endpoint de duas maneiras diferentes, dependendo do formato que precisa (HTML para a interface web ou JSON para consumo via API). 

A lógica de listagem e busca está centralizada no [PostsController](file:///home/thigas/projetos/blog/app/controllers/posts_controller.rb#L4-L11).

---

### 1. Via API (Retorno em JSON)
Se você estiver consumindo os dados através de uma aplicação front-end, script ou cliente HTTP (como Postman, Insomnia ou curl), utilize o formato `.json`.

* **Listar todas as notícias:**
  * **Endpoint:** `GET /posts.json`
  * **Exemplo de URL (local):** `http://localhost:3000/posts.json`
  * **Exemplo de requisição (cURL):**
    ```bash
    curl -X GET "http://localhost:3000/posts.json"
    ```

* **Buscar notícias por título:**
  * Você pode filtrar as notícias usando o parâmetro `query`.
  * **Endpoint:** `GET /posts.json?query=termo_da_busca`
  * **Exemplo de requisição (cURL):**
    ```bash
    curl -X GET "http://localhost:3000/posts.json?query=Inteligência"
    ```

#### Estrutura do Retorno (JSON)
Conforme definido em [_post.json.jbuilder](file:///home/thigas/projetos/blog/app/views/posts/_post.json.jbuilder), a resposta retornará um array de objetos com o seguinte formato:
```json
[
  {
    "id": 1,
    "title": "Título da Notícia",
    "description": "Descrição curta da notícia",
    "body": "Corpo completo do post...",
    "created_at": "2026-07-11T18:00:00.000Z",
    "updated_at": "2026-07-11T18:00:00.000Z",
    "url": "http://localhost:3000/posts/1.json"
  }
]
```

---

### 2. Via Navegador (HTML)
Para visualizar diretamente na interface web da aplicação:

* **Página Inicial / Lista Geral:**
  * A rota raiz do projeto (`/`) e a rota `/posts` renderizam a mesma página de listagem.
  * **Exemplo de URL:** `http://localhost:3000/` ou `http://localhost:3000/posts`

* **Busca via Query Parameter:**
  * Também é possível filtrar diretamente na página usando o parâmetro de busca.
  * **Exemplo de URL:** `http://localhost:3000/posts?query=Inteligência`

Criar uma **Skill** personalizada para o **Google Antigravity** permite que você ensine fluxos de trabalho específicos, regras de formatação de código ou automações diretamente para o seu agente de IA.

Aqui está o passo a passo de como criar uma Skill no seu projeto:

---

### 1. Entendendo onde a Skill deve ficar
Você pode criar Skills em dois escopos:
*   **Workspace (Específica do projeto)**: Salva em `<raiz-do-projeto>/.agents/skills/<nome-da-skill>/`.
*   **Global (Disponível em qualquer projeto)**: Salva em `~/.gemini/config/skills/<nome-da-skill>/`.

---

### 2. Estrutura de Arquivos
Uma Skill básica precisa de apenas uma pasta e um arquivo principal chamado `SKILL.md`. 
A estrutura recomendada é:

```text
minha-nova-skill/
├── SKILL.md                 # Arquivo principal de instruções (Obrigatório)
├── scripts/                 # (Opcional) Scripts auxiliares (Python, Shell, etc.)
├── examples/                # (Opcional) Exemplos práticos de uso/código
├── resources/               # (Opcional) Templates e recursos adicionais
└── references/              # (Opcional) Documentações e especificações longas
```

---

### 3. Criando o `SKILL.md`
O arquivo `SKILL.md` deve conter um **YAML Frontmatter** no topo (contendo `name` e `description`) e um corpo em Markdown com as instruções detalhadas.

Crie o arquivo com a seguinte estrutura:

```markdown
---
name: nome-da-sua-skill
description: Descreva de forma concisa e clara o que esta skill faz. (O agente usa essa descrição para saber quando ativá-la).
---

# Nome da Skill

> Uma breve introdução ou aviso importante sobre a Skill.

## 🎯 Quando usar esta Skill
Descreva os cenários onde o agente deve ativar esta skill (ex: quando o usuário pedir para gerar um relatório, publicar um artigo, rodar testes de integração).

## 🚀 Como Executar
Instruções passo a passo que o agente deve seguir:
1. Primeiro faça a ação A...
2. Valide com a ação B...
3. Use o template X localizado na pasta `resources/`...

## 📝 Padrões e Regras
Adicione restrições ou boas práticas (ex: sempre responder em português, manter arquivos com menos de 500 linhas, etc).
```

---

### 4. Como Ativar e Usar
1. **Descoberta Automática**: Se você criar a pasta dentro do diretório padrão `.agents/skills/` do seu projeto, o Antigravity irá **detectá-la e carregá-la automaticamente**. Não é necessário registrar manualmente.
2. **Execução**: Sempre que a descrição da sua skill der match com o que você solicitar no chat (ou se você chamar explicitamente a skill usando a barra `/nome-da-sua-skill`), o Antigravity lerá as instruções contidas no `SKILL.md` e as executará fielmente.

A perda de dados ao "subir" a aplicação Rails (quando usando SQLite3) geralmente ocorre por conta de como o ciclo de vida do banco de dados está sendo tratado no ambiente em que ela é executada.

Aqui estão os **3 motivos mais comuns** para isso acontecer e como resolver:

---

### 1. Você está rodando a aplicação via Docker localmente sem montar Volumes (Mais provável)
Se você estiver construindo a imagem do Docker e subindo o container manualmente com o comando padrão:
```bash
docker run -d -p 80:80 -e RAILS_MASTER_KEY=... --name notic_ia notic_ia
```
Os containers Docker são **efêmeros** por padrão. Isso significa que qualquer arquivo criado ou alterado dentro do container (como o arquivo do banco de dados em `/rails/storage/production.sqlite3`) é **destruído permanentemente** quando o container é parado e recriado (ao fazer um novo build ou deploy).

* **Como corrigir localmente**:
  Você precisa montar um volume persistente do Docker para que a pasta de armazenamento seja salva no seu disco rígido (host) e não dentro da camada efêmera do container:
  ```bash
  docker run -d -p 80:80 -v notic_ia_storage:/rails/storage -e RAILS_MASTER_KEY=... --name notic_ia notic_ia
  ```

---

### 2. O Banco de Dados de Desenvolvimento está sendo recriado
Se você está rodando o projeto em sua máquina de desenvolvimento local usando o comando `bin/dev` ou `rails server` tradicional, os dados são salvos em [database.yml](file:///home/thigas/projetos/blog/config/database.yml) no arquivo físico `storage/development.sqlite3`.
* Se você estiver usando um ambiente containerizado temporário (como *Gitpod*, *GitHub Codespaces* ou *Dev Containers*) e o container de desenvolvimento for recriado, a pasta `storage/` pode não estar sendo persistida no host.
* Certifique-se de que não há nenhum script de inicialização ou comando (como `rails db:reset` ou `rails db:migrate:reset`) rodando automaticamente na sua rotina de inicialização, pois esses comandos deletam e recriam o banco de dados do zero.

---

### 3. Deploy em Produção sem Mapeamento Correto
Caso esteja subindo a aplicação em produção com o **Kamal** (usando o comando `kamal deploy`):
O seu arquivo [deploy.yml](file:///home/thigas/projetos/blog/config/deploy.yml) já está configurado corretamente com o volume persistente:
```yaml
volumes:
  - "notic_ia_storage:/rails/storage"
```
Isso garante que o Kamal crie um volume no Docker do servidor de produção chamado `notic_ia_storage` e o monte em `/rails/storage`. 
* Se você estiver perdendo os dados no servidor, verifique se o deploy está sendo feito estritamente através do Kamal. Se você deletar o volume do Docker no servidor (`docker volume rm notic_ia_storage` ou `docker volume prune`), a base de dados do SQLite será apagada.

O motivo de você estar perdendo os dados a cada novo deploy com o Kamal é que o seu arquivo de configuração estava usando um **Volume Nomeado do Docker** (`notic_ia_storage:/rails/storage`). 

### Por que isso causava a perda de dados?
Toda vez que o Kamal realiza um novo deploy:
1. Ele baixa a nova imagem e **sobe novos containers** com a tag/versão atualizada (ex: `notic_ia-web-versao123`).
2. Ele interrompe e remove os containers antigos. Durante esse processo de limpeza (`docker container prune` e `docker volume prune`), o Docker muitas vezes **desassocia ou destrói volumes nomeados temporários** que não estejam explicitamente travados, fazendo com que o arquivo do SQLite (`production.sqlite3`) seja apagado.

---

### A Solução
A recomendação oficial para usar SQLite com o Kamal é usar um **Bind Mount** apontando para um **caminho absoluto no disco rígido do próprio servidor (host)**, em vez de um volume do Docker. Dessa forma, a pasta física existe no servidor fora do ecossistema do Docker e nunca é mexida pelos deploys.

Eu já alterei o mapeamento de volume no seu [config/deploy.yml](file:///home/thigas/projetos/blog/config/deploy.yml) e enviei a atualização para o seu GitHub na branch `main`:

```yaml
# Antes:
volumes:
  - "notic_ia_storage:/rails/storage"

# Agora (Corrigido):
volumes:
  - "/var/lib/notic_ia/storage:/rails/storage"
```

---

### ⚠️ O que você precisa fazer agora (Passo a Passo)

Para que o deploy funcione sem dar erros de permissão de escrita, você precisa **criar a pasta no seu servidor de hospedagem** e dar permissão para o usuário do Rails (UID `1000`) poder escrever nela. 

Acesse o seu servidor via **SSH** e rode os seguintes comandos:

```bash
# 1. Crie a pasta física de armazenamento no disco do servidor
sudo mkdir -p /var/lib/notic_ia/storage

# 2. Defina o usuário 1000:1000 (o usuário 'rails' do container) como dono da pasta
sudo chown -R 1000:1000 /var/lib/notic_ia/storage

# 3. Garanta as permissões corretas de leitura e escrita
sudo chmod -R 775 /var/lib/notic_ia/storage
```

Depois de rodar esses comandos no servidor, você pode fazer o deploy novamente com:
```bash
kamal deploy
```

A partir deste momento, todos os seus posts e dados do SQLite persistirão permanentemente na pasta `/var/lib/notic_ia/storage` do servidor e **nunca mais serão perdidos nos deploys**!

Por padrão, os serviços web (Web Services) do Render usam um **sistema de arquivos efêmero (stateless)**. Toda vez que você faz uma alteração no código ou o container reinicia:
1. O Render **destrói** o container antigo por completo (junto com tudo o que foi gravado nele, incluindo o arquivo do SQLite em `storage/production.sqlite3`).
2. Ele cria um container **novo e limpo** do zero com base na nova imagem gerada do seu Dockerfile.

---

### A Solução: Render Disks (Persistent Disks)

Para resolver isso, você precisa anexar um **disco rígido persistente** ao seu serviço web no Render. Esse disco fica montado no servidor de forma externa ao container e sobrevive a qualquer deploy ou reinicialização.

Eu já atualizei o seu arquivo [config/database.yml](file:///home/thigas/projetos/blog/config/database.yml) para usar **caminhos absolutos** (`/rails/storage/...`) no ambiente de produção e acabei de enviar a alteração para a branch `main` do seu GitHub.

Agora, siga este passo a passo rápido no painel do Render:

#### 1. Adicione o Disco no Painel do Render
1. Acesse o painel da sua conta no **Render.com**.
2. Clique no seu **Web Service** (`notic-ia`).
3. No menu lateral esquerdo, vá em **Disks**.
4. Clique em **Add Disk** e configure da seguinte forma:
   - **Name**: `notic-ia-storage` (ou qualquer nome curto)
   - **Mount Path**: `/rails/storage` *(Esse caminho é crucial! É onde o seu container espera encontrar o banco)*
   - **Size**: `1 GiB` *(O plano básico de 1 GB custa apenas $0.25/mês ou é grátis dependendo do seu plano, e é suficiente para milhares de postagens de texto)*
5. Clique em **Save** ou **Create**.

#### 2. Faça um novo Deploy
Uma vez adicionado o disco, o Render reiniciará a sua aplicação para montá-lo:
* No primeiro boot com o disco montado, o diretório `/rails/storage` estará vazio.
* O entrypoint do Rails (`bin/docker-entrypoint`) detectará que o banco de dados não existe no disco e criará o arquivo `production.sqlite3` do zero, rodando automaticamente todas as migrações de banco.
* A partir de agora, **todos os deploys futuros não apagarão seus dados**, pois o banco estará seguro no Render Disk!

Aqui estão as **3 melhores alternativas** para o seu caso:

---

### Alternativa 1: Migrar para um banco PostgreSQL na Nuvem (Altamente Recomendado)
Em vez de salvar um arquivo SQLite local, a sua aplicação se conecta a um banco de dados PostgreSQL externo. A aplicação se torna 100% "stateless" (sem arquivos importantes locais), o que significa que o Render pode destruir e recriar o container no deploy à vontade, pois os dados estão salvos em segurança em outro servidor.

*   **Onde conseguir um banco PostgreSQL grátis**:
    *   **Neon (neon.tech)**: Oferece um banco PostgreSQL moderno, rápido e com plano gratuito permanente excelente para projetos e blogs.
    *   **Supabase (supabase.com)**: Oferece bancos PostgreSQL gratuitos muito robustos.
    *   **Render PostgreSQL**: O Render oferece um banco de dados interno, mas o plano gratuito dele expira após 90 dias.
*   **Como configurar**:
    1. Adicionar `gem "pg"` ao `Gemfile`.
    2. Atualizar o `config/database.yml` de produção para usar o adaptador do Postgres:
       ```yaml
       production:
         primary:
           adapter: postgresql
           url: <%= ENV["DATABASE_URL"] %>
       ```
    3. Adicionar a variável de ambiente `DATABASE_URL` no painel do Render com a URL de conexão que o Neon/Supabase te der (ex: `postgres://usuario:senha@host/banco`).

---

### Alternativa 2: Turso (SQLite na Nuvem)
Se você gosta muito da velocidade do SQLite e quer mantê-lo, o **Turso (turso.tech)** é um serviço que hospeda bancos de dados SQLite (usando o libSQL) na nuvem.
*   O plano gratuito é extremamente generoso (permite até 9GB de dados).
*   Você instala a gem `activerecord-libsql-adapter` no seu Rails.
*   Configura a conexão em produção apontando para a URL do Turso (ex: `libsql://noticia.turso.io`) e um Token de acesso. O banco de dados deixa de ser um arquivo físico local e passa a rodar na nuvem do Turso.

---

### Alternativa 3: Litestream (Backup automático do SQLite para S3 / Cloudflare R2)
O **Litestream** é uma ferramenta open-source que roda dentro do seu container Docker junto com o Rails. 
*   Toda vez que o SQLite sofre uma escrita, o Litestream envia a alteração em tempo real para um bucket S3 (como o **Cloudflare R2**, que é 100% gratuito para até 10GB de dados).
*   Quando o Render cria um novo container no deploy, o Litestream detecta, baixa o banco de dados atualizado do R2 para dentro do container e o Rails inicia com todos os posts intactos.
*   *Desvantagem*: Requer alterar o `Dockerfile` para instalar o Litestream e configurar variáveis de acesso ao S3 no Render.

---

A **Alternativa 1 (PostgreSQL externo no Neon ou Supabase)** é a mais comum e madura no ecossistema Rails e a mais fácil de configurar.

   - Para que o seu servidor **MCP** e scripts locais continuem postando sem problemas, adaptamos o concern `Authentication` para responder a requisições JSON validando um Token de API (`Authorization: Bearer <token>`) configurado em suas credentials encriptadas.

   - Redesenhamos a tela de login (`/session/new`) e redefinição de senha para seguirem o padrão visual clean de leitura do blog.
